KV Cache 的工作集与雪崩
1968 年的理论如何解释 LLM 推理

Prefill 节点的 HBM 里塞着许多 session 的 KV cache。谁该留、留多久、超卖了会发生什么——这组问题和 1968 年“给每个进程多少内存”的问题同构。本页只讲四个关键点。

映射

1. 从分页内存到 KV Cache：一张对照表

只考虑 prefill 节点：一个 session 是多轮对话，每轮到达时追加一段输入，计算这一轮时要对全部历史做 attention。对照 1968 年的世界：

一个先说清的澄清：miss 不是卡死，是重算——token 文本就是“后备存储”，任何 KV 都能重新算出来，只是贵。

关键点一

2. 一个 session 需要什么：全部前缀

关键事实　计算任何一轮都要对全部历史做 attention——所以一个 session 的“工作集”就是它的整个前缀 KV：全有或全无，逐轮只增不减。

关键点二

3. Miss 有多贵：99% 命中率是不够的

KV 世界的惩罚是惩罚比 R = context 长度 ÷ 新一轮长度：5 万 token 上下文的 session 发来 500 token 的一轮，命中只算 500，全 miss 要算 50500——R = 100 很日常，长文档场景轻松上千。单 session 的计算效率（有效计算占比）是：

e = 1 / (1 + (1 − h) × R)　　其中 h 为命中率，R = context / 轮长

R = 100 时，h = 99% 只换来 50% 的效率；h = 90% 只剩 9%——miss 率必须远小于 1/R。这就是为什么 KV cache 的命中率指标动辄要求 99.x%。

关键点三（上）

4. 把 5 个会话放进一个 LRU 池：stack distance

第 2 节的账单来自下面这个池子。池子的规则只有一条（vLLM / SGLang 的真实行为）：块进来就留下；需要空间时，从“最久没说话”的会话尾部逐出。在这条规则下，每个回头轮的命运由一个数决定：

Stack distance = 自上次说话以来别人垫进池子的量 + 自己的前缀。柱子低于容量线 M → 前缀全在，免费复用；柱子高出 M 的部分（变红）= 这一轮要重算的量，最多红到整个前缀。

柱高与 M 无关——拖动 M，命中与重算实时翻转，柱子永远不动：一张图回答所有容量。

关键点三（下）

5. 雪崩线：跨过去就是断崖

上一节的柱子高矮不一，是因为 5 个会话节奏各异——有人插空、有人沉默。现在把负载推到极限：N 个 session 轮流说话、context 都是 c̄、每轮新增 0.5k token。这时每根柱子都一样高：stack distance = 别人垫的 (N−1)·c̄ + 自己的 c̄ = N·c̄。柱子同高，结局就只有两种：整齐地全在 M 线下（人人全命中），或整齐地全超线（人人重算）——命中率不是缓降而是阶跃归零。悬崖的位置就在：

雪崩线：N × c̄ = M（即 Σcontext = M）　——HBM 越大养得起的 session 越多，跨过线即雪崩。

模型公式与推导

块级 LRU：每个回头轮的 stack distance = N·c̄，重算量 = clamp(N·c̄ − M, 0, c̄)（第 4 节的规则），故命中率 h = clamp((M − (N−1)·c̄) / c̄, 0, 1)——在宽度仅 c̄ 的过渡带内从 1 跌到 0。每轮实际算量 = δ + (1−h)·c̄，goodput = δ / (δ + (1−h)·c̄)。

session 级准入：K = ⌊M / c̄⌋ 个 session 驻留、全命中；其余 N−K 个不驻留，每轮自付全量重算，但不侵占驻留者的空间。goodput = N·δ / (K·δ + (N−K)·(δ + c̄))。

准入控制消灭不了工作量（超额 session 开口就得算，绿线也会下垂），它消灭的是互相踩踏：驻留的 K 个 session 效率恒为 100%，损失被隔离在超额者身上。

比前篇更险恶的一点：雪崩发生时 GPU 利用率是 100%——它在满负荷地重算历史。OS 抖动的体征是 CPU 空转，一眼可诊断；KV 雪崩在利用率面板上一片健康，必须盯 goodput、命中率、TTFT 才看得见。

而且雪崩会自我放大，不需要任何人犯错：每轮变慢 → 在途请求堆积 → 池子更挤；TTFT 超时 → 客户端重试 → 同一个长 prefill 提交两遍。1968 年那个火上浇油的调度器，今天叫超时重试。

药方与要点

6. 药方还是 1968 年那一个

① Σcontext ≤ M 是生死线

一个 session 的需求 = 全部前缀，只增不减。session 数不变，需求也会自己爬过线——盯住总和，别盯个数。

② R ≈ 100 让 99% 命中率只值 50% 效率

miss 的代价是重算整个历史。miss 率必须远小于 1/R——不是洁癖，是效率公式的暴政。

③ 雪崩是隐形的

崩溃时 GPU 利用率是 100%——它在满负荷做无用功。报警要盯 goodput、命中率、TTFT，别信利用率。

④ 解法 = 准入控制

超线时让新轮次在门口排队，别放进池子互踩；逐出从 session 尾部开始；把 session 路由回缓存所在的节点。
