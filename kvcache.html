经典理论的现代转世

KV Cache 的工作集与雪崩
1968 年的理论如何解释 LLM 推理

Prefill 节点的 HBM 里塞着许多 session 的 KV cache，谁该留、留多久、超卖了会发生什么——这组问题和 1968 年"给每个进程多少内存"的问题惊人地同构，又在几个关键地方彻底不同。本页沿用前篇的路线：先看清工作集，再走到雪崩边缘。

映射

1. 从分页内存到 KV Cache：一张对照表

只考虑 prefill 节点：一个 session 是一条多轮对话轨迹，每轮到达时追加一段输入；计算这一轮时要对全部历史做 attention，所以该 session 的整个前缀 KV 必须驻留在 HBM 中。对照 1968 年的世界：

三个必须先说清的澄清：

① Miss 不是卡死，是重算。token 文本本身就是“后备存储”——任何 KV 都能从文本重新算出来，只是贵。所以 KV cache 严格说不是请求调页，而是物化视图：数据永远可再生，问题只是你愿不愿意付这个 GPU 账单。真正的三选一是：命中（便宜）、重算（贵，且要挤占别人的空间）、排队（延迟）。

② 部分前缀有部分价值，但只能从尾巴逐出。保留最长连续前缀就能省下等长的重算（vLLM / SGLang 的 prefix cache 正是这么做的）；可要是从中间挖走一块，洞之后的所有 block 实际上都废了。前篇“逐出完整工作集、不要人人剃一刀”的教训，在这里的形态是：逐出永远从 session 的尾部开始。

③ 工作集原理在轮内是被硬件强制的。Denning 要靠调度策略保证“工作集不完整就不许运行”；而 attention 在数学上就要求全前缀同时驻留——原理被直接编译进了算法。于是所有策略自由度都落在轮与轮之间：给谁保留、保留多久、新轮次来了排队还是付费重算。这正是下面几节的全部内容。

概念一的转世

2. Session 的工作集：谁的 KV 该在 HBM 里

定义　时刻 t、窗口 τ 下的工作集 W(t, τ)：最近 τ 秒内有过轮次到达的 session，它们全部前缀 KV 的并集。所需 HBM 就是这些 session 的 context 长度之和。

与前篇的关键区别：session 内部没有局部性可言——每轮都要摸全部历史 block，块级访问是“循环全扫”。所有的局部性都在 session 粒度的时间维度上：刚说过话的 session 大概率马上还会说话（用户还在对话界面前），沉默越久回来的概率越低。这里的“轮次间隔”（分时系统文献里的老词 think time，指用户读回复、打字的时间，与模型的 thinking 无关）就是前篇引用串局部性的转世。

① 上图的一个点 = 一轮输入的到达。它标记的是“这个 session 在这个时刻说了句话”，不是内存块。KV block 在下图里：一个 session 那条横带的厚度，才是它全部前缀 KV 的量。

② 窗口 = “最近 τ 秒”这个时间段——蓝色竖带，右边缘是当前时刻 t，宽度是 τ。与前篇是同一个概念，只是单位从“最近 τ 次访存”换成了“最近 τ 秒”。

③ 窗口筛的是 session，不是块。一行里只要有至少一个点落进窗口，这个 session 就算活跃（整行高亮）→ 它的全部前缀 KV 必须驻留 → 下图当前时刻竖线上的蓝色剖面，就是此刻驻留的各个 session 各占多厚（挤到容量线 M 之上的部分变红）。前篇的窗口是逐页筛选的；这里一旦选中就是全量驻留——因为 attention 需要完整前缀。

④ τ 管的是“沉默期间留不留”，不是“算的时候给不给”。计算某一轮的那一刻，KV 必须全量在 HBM——谁也省不掉，所以每个点到来的瞬间，它的带子一定出现在堆里。τ 决定的是这轮算完之后、下一轮到来之前，这块内存要不要继续占着。把 τ 拉小试试：带子在每轮之后很快消失，HBM 省了，但沉默超过 τ 的回头轮次要付全量重算（内存照样临时全量分配，只是内容得现算，成本贵 R 倍，见第 4 节）；τ→0 就是“算完即丢”，每轮冷启动。留 = 用 HBM 换掉未来的重算；丢 = 用未来的重算换出 HBM，τ 是这笔投资的持有期。

值得盯着看的三个现象：

① 会话 2 在 46 秒后永远沉默——用户关掉了网页。但系统看到的和一次长停顿一模一样：死亡是不可观测的。前篇里旧页面会随局部性阶段切换自然滑出窗口；这里没有阶段切换，唯一的“自然过期”就是沉默超过 τ。窗口机制从一种优化手段变成了逻辑上的必需品。

② 会话 3 停顿 60 秒后回来了——τ 小于停顿时长时它已被逐出，回来那一轮要全量重算。τ 的两难和前篇完全平行：太小误杀还会回来的 session，太大让死人占着 HBM。

③ 带子只增不减。单个 session 活着时 context 单调增长（没有收缩机制），所以同样的并发数下，抖动压力随 session 年龄悄悄上升——负载没变，系统也会滑向红区。

参数选择

3. τ 怎么选：需求曲线与回头 miss 率

前篇里 τ 的选择靠 s(τ) 曲线的膝点，且缺页率恰好是它的斜率（m(τ) = s′(τ)）。这个恒等式在这里有直接的转世：把保留窗口定为 τ，则 miss 率 ≈ 轮次间隔超过 τ 的概率——s(τ) 花的是 HBM，m(τ) 省的是重算，两条曲线对同一份轨迹计算，夹出 τ 的合理区间：

定量基础

4. Miss 有多贵：99% 命中率是不够的

前篇的支点是"缺页服务时间比一次访存慢 4 个数量级"。这里的对应物不是常数，而是一个随 session 变老不断增大的比值：惩罚比 R = context 长度 ÷ 新一轮长度。5 万 token 上下文的 session 发来 500 token 的一轮：命中时只算 500，全 miss 时要算 50500——R = 100 很日常，长文档场景轻松上千。于是单 session 的计算效率（有效计算占比）是：

e = 1 / (1 + (1 − h) × R)　　其中 h 为命中率，R = context / 轮长

概念二的转世

5. 雪崩线：HBM、session 数与 context 的三角关系

现在把 N 个 session 放进同一块 HBM。每个 session 平均 context 为 c̄，每轮新增 δ = 0.5k token。总需求 N·c̄ ≤ M 时人人全命中；一旦超过，块级 LRU 会遭遇缓存理论里的经典病理——循环扫描略大于缓存的集合，命中率不是缓降而是阶跃归零（每个 session 每轮全扫自己的前缀，回来时自己的块恰好是最旧的，刚好被挤光；数据库 buffer pool 的 scan thrashing 同款）。前篇的悬崖需要一个教学用的幂指数，这里的悬崖是结构性的，位置就在：

雪崩线：M = N × c̄　——HBM 越大养得起的 session 越多（线的斜率就是 c̄），跨过线即雪崩。

两条线在跨线后都会下坠（超额 session 的重算是真实工作量，谁也变不走），但读法完全不同：LRU 下所有 session 一起沉没；准入控制下驻留的 K 个 session 各自仍是 100% 效率，损失被隔离在超额者身上——M=400、c̄=50 时，N=9 的总 goodput 是 8.3% vs LRU 的 1%，差 8 倍，且随 N 增大持续领先（见数据表）。

模型公式与推导

块级 LRU：session 回来时，缓存里是比它更新的 N−1 个 session 的 context，留给它的空间是 M − (N−1)·c̄。命中率 h = clamp((M − (N−1)·c̄) / c̄, 0, 1)——在宽度仅 c̄ 的过渡带内从 1 跌到 0。每轮实际算量 = δ + (1−h)·c̄，goodput = δ / (δ + (1−h)·c̄)。

session 级准入：K = ⌊M / c̄⌋ 个 session 驻留、全命中；其余 N−K 个不驻留，每轮自付全量重算，但不侵占驻留者的空间。goodput = N·δ / (K·δ + (N−K)·(δ + c̄))。

注意与前篇的不同：OS 里被挂起的进程不消耗 CPU，绿线是平的；这里超额 session 开口说话就得算，绿线也会下垂——准入控制不能消灭工作量，它消灭的是互相踩踏：驻留的 K 个 session 效率恒为 100%，损失被隔离在超额者自己身上。若在门口直接排队/限流（不处理超额轮次），goodput 可保持 ~100%，代价是排队延迟——那才是前篇"整体挂起"的严格对应物。

比前篇更险恶的一点：雪崩发生时 GPU 利用率是 100%。OS 抖动的体征是 CPU 空转、磁盘狂响，一眼可诊断；KV 雪崩时监控面板一片健康——GPU 在满负荷地重算历史。必须换指标才能看见它：goodput、cache 命中率、TTFT，缺一不可。

正反馈螺旋的现代转世：重试风暴。Denning 的螺旋是"CPU 空闲 → 调度器放入更多进程"；这里的版本更凶：雪崩 → TTFT 暴涨 → 客户端超时重试 → 同一个长 prefill 被提交两遍 → 需求进一步过载。1968 年那个火上浇油的调度器，今天叫超时重试策略。

药方

6. 药方：还是工作集原理，但形态变了

前篇的药方是一句话：Σ|Wi| ≤ M，装不下就整体挂起。转世到 KV cache，它裂解成一组互相配合的机制：

用 τ 管理保留（本页第 2、3 节）

沉默超过 τ 的 session 整体让位——既是逐出策略也是死亡推断。τ 对着轮次间隔分布的膝点选，工程形态就是 KV cache 的 TTL。

Session 粒度逐出，尾部优先

要让位就整条带子抽走（或从尾部截断），绝不从中间挖洞、绝不让新 session 零敲碎打地踩别人的前缀——那是雪崩的引信。

门口准入，而非池内互踩

需求过线时，宁可让新轮次排队/限流（付延迟），也不放进来触发全池重算（付吞吐）。vLLM 的调度器不见 block 不排批、满池时整序列 preempt（recompute 或 swap 二选一——正是"重算 vs 换页"），都是这个原则。

亲和路由：别弄丢工作集

多个 prefill 节点时，把 session 路由回持有其缓存的节点（对应 OS 的进程迁移代价）。SGLang 的 radix tree 前缀共享、Mooncake 的 KVCache 中心调度，都是把"工作集在哪"当成一等公民的调度。

还有一个前篇没有的对手值得点名：共享前缀。公共 system prompt 是所有 session 工作集的交集——1968 年共享代码段的转世，也是现代 prefix cache 命中率的最大来源。它不改变雪崩的数学，但改变了 c̄ 的账。

收尾

7. 像与不像：六个本质差异

① 局部性的单位变了

页 → session 前缀。session 内部每轮全扫历史，没有块级局部性；所有局部性都在"谁最近说过话"这个维度上。

② 工作集只增不减，死亡不可观测

进程工作集随阶段漂移；session 前缀单调生长直到无声无息地死去。τ 窗口从优化手段变成推断死亡的唯一途径。

③ 后备存储是"重算"

缺页读磁盘，miss 付 GPU：数据永远可再生，代价 ∝ 丢失量。KV cache 是物化视图，不是请求调页。

④ 惩罚比随年龄增长

T/访存 ≈ 10⁴ 是物理常数；R = context/轮长逐轮变大。负载不变，系统也会滑向悬崖。

⑤ 雪崩是隐形的

OS 抖动 CPU 空转，可见；KV 雪崩 GPU 100% 忙于无用功。诊断指标必须换成 goodput / 命中率 / TTFT。

⑥ 悬崖更陡，且无需拟合

前篇的幂指数是教学选择；这里"循环全扫 + LRU"的阶跃归零是可证明的病理，雪崩线 M = N·c̄ 是结构性的。
