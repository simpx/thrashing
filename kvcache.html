经典理论的现代转世

KV Cache 的工作集与雪崩
1968 年的理论如何解释 LLM 推理

Prefill 节点的 HBM 里塞着许多 session 的 KV cache。谁该留、留多久、超卖了会发生什么——这组问题和 1968 年“给每个进程多少内存”的问题同构。本页只讲四个关键点，一节一个。

映射

1. 从分页内存到 KV Cache：一张对照表

只考虑 prefill 节点：一个 session 是多轮对话，每轮到达时追加一段输入，计算这一轮时要对全部历史做 attention。对照 1968 年的世界：

CPU 利用率Goodput：GPU 算力中花在“新 token”上的比例

两个先说清的澄清：① miss 不是卡死，是重算——token 文本就是“后备存储”，任何 KV 都能重新算出来，只是贵；② 逐出只能从尾部开始——保留最长连续前缀就能省下等长的重算，但从中间挖洞，洞之后的块全部作废。

关键点一

2. 一个 session 需要什么：全部前缀

关键事实　计算任何一轮都要对全部历史做 attention——所以一个 session 的“工作集”就是它的整个前缀 KV：全有或全无，逐轮只增不减。前篇的工作集逐页进出窗口；这里没有那种弹性。

关键点二

3. Miss 有多贵：99% 命中率是不够的

前篇的支点是“缺页服务时间比一次访存慢 4 个数量级”。这里的对应物是惩罚比 R = context 长度 ÷ 新一轮长度：5 万 token 上下文的 session 发来 500 token 的一轮，命中只算 500，全 miss 要算 50500——R = 100 很日常，长文档场景轻松上千。单 session 的计算效率（有效计算占比）是：

e = 1 / (1 + (1 − h) × R)　　其中 h 为命中率，R = context / 轮长

关键点三（上）

4. 把 9 个 session 放进一个 LRU 池

现在回答“第 2 节里是谁挤掉了缓存”。池子的规则只有一条（vLLM / SGLang 的真实行为）：块进来就留下；只有新的分配需要空间时，才从“最久没说话”的 session 的尾部逐出。第 2 节的账单图，就是从下面这张全景里抽出一个 session、把它的每个点展开成一行。

① 一个点 = 一轮到达（画在它自己 session 的带子里）：蓝 = 前缀全在，免费复用；红 = 沉默期间被挤过，本轮要付第 3 节的 R 倍账单。

② 带子 = 驻留的块，被削顶 = 正在被逐出（凶手是同一时刻长高的另一条带子）；虚线 = 内存无限时本可缓存的总量，它与堆顶之间的浅红楔子 = 已被挤出的块——谁回来谁付账。

LRU 池全景：轮次（点）、驻留（带）、逐出（空缺与削顶）、容量（横线）

distance 轴下的几何事实：每条带子在最后一个点之后的尾长都 ≈ M。“一段缓存能活多久”与“容量有多大”，在这个坐标系里是同一个数字。

t = 　池中 session 　已用 / 容量 　已挤出

三件值得盯着看的事：会话 2 死了，但它的带子赖到被挤才走——LRU 赌“最老的最可能已死”，赌对免费、赌错付 R 倍；会话 3 停顿 61 秒回来付了全额账单——把 M 拉大到需求线之上，同一个点变蓝：miss 是容量的错，不是它的错；带子只增不减——session 数不变，需求也在自己爬坡。

关键点三（下）

5. 雪崩线：跨过去就是断崖

上一节的池子只有 9 个 session；现在把负载推到极限。设 N 个 session 平均 context 为 c̄、每轮新增 δ = 0.5k token。总需求 N·c̄ ≤ M 时人人全命中；一旦超过，LRU 池遭遇缓存理论的经典病理——循环扫描略大于缓存的集合，命中率不是缓降而是阶跃归零（每个 session 每轮全扫自己的前缀，回来时自己的块恰好最旧、刚好被挤光）。前篇的悬崖需要一个教学用的幂指数；这里的悬崖是结构性的，位置就在：

雪崩线：M = N × c̄　——HBM 越大养得起的 session 越多（斜率就是 c̄），跨过线即雪崩。

模型公式与推导

块级 LRU：session 回来时，缓存里是比它更新的 N−1 个 session 的 context，留给它的空间是 M − (N−1)·c̄。命中率 h = clamp((M − (N−1)·c̄) / c̄, 0, 1)——在宽度仅 c̄ 的过渡带内从 1 跌到 0。每轮实际算量 = δ + (1−h)·c̄，goodput = δ / (δ + (1−h)·c̄)。

session 级准入：K = ⌊M / c̄⌋ 个 session 驻留、全命中；其余 N−K 个不驻留，每轮自付全量重算，但不侵占驻留者的空间。goodput = N·δ / (K·δ + (N−K)·(δ + c̄))。

与前篇的不同：OS 里被挂起的进程不消耗 CPU，绿线是平的；这里超额 session 开口说话就得算，绿线也会下垂——准入控制消灭不了工作量，它消灭的是互相踩踏：驻留的 K 个 session 效率恒为 100%，损失被隔离在超额者身上。若在门口直接排队/限流，goodput 可保持 ~100%，代价是排队延迟——那才是前篇“整体挂起”的严格对应物。

比前篇更险恶的一点：雪崩发生时 GPU 利用率是 100%——它在满负荷地重算历史。OS 抖动的体征是 CPU 空转，一眼可诊断；KV 雪崩在利用率面板上一片健康，必须盯 goodput、命中率、TTFT 才看得见。

而且雪崩会自我放大，不需要任何人犯错：每轮变慢 → 在途请求堆积（Little 定律）→ 池子更挤；TTFT 超时 → 客户端重试 → 同一个长 prefill 提交两遍。1968 年那个火上浇油的调度器，今天叫超时重试。

药方与要点

6. 药方还是 1968 年那一个

① Σcontext ≤ M 是生死线

一个 session 的需求 = 全部前缀，只增不减。session 数不变，需求也会自己爬过线——盯住总和，别盯个数。

② R ≈ 100 让 99% 命中率只值 50% 效率

miss 的代价是重算整个历史。miss 率必须远小于 1/R——不是洁癖，是效率公式的暴政。

③ 雪崩是隐形的

崩溃时 GPU 利用率是 100%——它在满负荷做无用功。报警要盯 goodput、命中率、TTFT，别信利用率。

④ 解法 = 准入控制

超线时让新轮次在门口排队，别放进池子互踩；逐出从 session 尾部开始；把 session 路由回缓存所在的节点。vLLM 的 preemption、SGLang 的 radix cache、Mooncake 的调度，都是这一条的工程形态。
