经典理论的现代转世

KV Cache 的工作集与雪崩
1968 年的理论如何解释 LLM 推理

Prefill 节点的 HBM 里塞着许多 session 的 KV cache。谁该留、留多久、超卖了会发生什么——这组问题和 1968 年“给每个进程多少内存”的问题同构。本页只讲四个关键点。

映射

1. 从分页内存到 KV Cache：一张对照表

只考虑 prefill 节点：一个 session 是多轮对话，每轮到达时追加一段输入，计算这一轮时要对全部历史做 attention。对照 1968 年的世界：

一个先说清的澄清：miss 不是卡死，是重算——token 文本就是“后备存储”，任何 KV 都能重新算出来，只是贵。

关键点一

2. 一个 session 需要什么：全部前缀

关键点二

3. Miss 有多贵：99% 命中率是不够的

KV 世界的惩罚是惩罚比 R = context 长度 ÷ 新一轮长度：5 万 token 上下文的 session 发来 500 token 的一轮，命中只算 500，全 miss 要算 50500——R = 100 很日常，长文档场景轻松上千。单 session 的计算效率（有效计算占比）是：

每轮实际要算的 = 新增 δ ＋ 丢掉的前缀 (1−h)·C，其中只有 δ 是有效工作（缓存完美时也得算）。有效 ÷ 实际：

e = δ / (δ + (1−h)·C) = 1 / (1 + (1 − h) × R)　　h = context token 的命中率，R = C / δ

R = 100 时，h = 99% 只换来 50% 的效率（实际算 500+500=1000，里面只有 500 是新活）；h = 90% 只剩 9%——miss 率必须远小于 1/R。这就是为什么 KV cache 的命中率指标动辄要求 99.x%。而 h = 0 的极限 1/(1+R)，就是第 5 节悬崖的谷底。

关键点三（上）

4. 把 5 个会话放进一个 LRU 池

第 2 节的账单来自下面这个池子。池子的规则只有一条（vLLM / SGLang 的真实行为）：块进来就留下；需要空间时，从“最久没说话”的会话尾部逐出。块级 LRU 落在只增不减的前缀上，涌现出来的其实是 session 级 LRU：所有会话按“最近说话”排成一个栈——谁开口谁浮到栈顶，容量线以下的尾巴已被逐出。

Stack distance = 回来那一刻你在栈里的深度（压在上面的别人 + 自己的前缀）。深度 ≤ M → 前缀全在，免费复用；深度超出 M 的部分 = 已被逐出的尾巴，本轮重算，最多亏掉整个前缀。

栈的顺序与 M 无关——拖动容量线，只有“命中还是重算”在翻转，栈本身一格不动：一遍回放，回答所有容量（Mattson 1970 的老技巧）。

关键点三（下）

5. 雪崩线：跨过去就是断崖

上一节小曲线的膝点，就是那 5 个会话的雪崩线——节奏参差，所以膝点是钝的。现在看最坏情况：N 个会话一样胖（context 都是 c̄、每轮新增 δ）、轮流说话。悬崖三步推出来：

① 回来时的深度人人都是 N·c̄——沉默期间其余 N−1 家各说了一句、各压上 c̄，加上自己的 c̄。

② 深度相同 → 命运相同——N·c̄ ≤ M 人人全命中；一旦超出，人人被啃掉同样长的尾巴 (N·c̄ − M)；超出达到一个会话宽（c̄）时，啃的就是整个前缀。为什么被啃光的每次都恰好是回来的人？LRU 挤"最久没说话的"，而轮流说话时，最久没说话的 = 下一个要开口的——逐出永远精确打中下一个请求（LRU 撞上循环访问的经典病理）。

③ 所以雪崩线在 x = Σcontext ÷ M = 1，谷底 = 全额重算时的 δ/(δ+c̄) = 1/(1+R)——第 3 节 h = 0 的那条公式。

模型公式与推导

准静态近似：context 的增长远慢于轮转，把每个会话的 context 记作常数 c̄、每轮新增 δ，R = c̄ / δ = 100。雪崩线 N* = M / c̄ = 10，即 x = Σcontext ÷ M = 1。

LRU 超卖：N ≤ N* 人人全命中。从 N* + 1 起，回来时留给你的空间 M − (N−1)·c̄ ≤ 0——整个前缀已被啃光，每轮算 δ + c̄，goodput 触底 1/(1+R)。过渡带恰好一个会话宽；改 R 只改谷底深浅。

session 级准入：K = N* 个会话驻留、恒 100% 命中；超额者每轮自付全量重算 → goodput = N·δ / (K·δ + (N−K)·(δ + c̄))。它同样下垂——准入消灭不了超额的工作量，它消灭的是互相踩踏：损失被隔离在超额者身上，驻留者毫发无损。

跨线不一定要加会话：turn 变多、context 自己长，也会把 Σcontext 顶过 M——第 4 节的回放就是这么崩的（5 个会话从头到尾没多没少，时序图的红圈却全堆在后半场）。区别只在速度：加会话是跳崖，长 context 是滑坡——过线后每转一圈人人多重算 N·δ，一两圈 goodput 就掉一个数量级。而且药方也在漂移：R 随 context 变大、谷底更深，准入养得起的 K = M/c̄ 在缩水——准入线必须随需求重算，设一次静态值不管用。下面让 5 个开局迥异的会话一直聊下去，看终局：

5 个会话的起点、增速、节奏各不相同；竖虚线 = Σcontext 爬过 M 的时刻。拖大 M，红色边界右移——但永远等在右边。

不管开局怎样，终局都一样：过线之后绿色一列列熄灭，最后人人全红——每轮重算全部历史，池子退化成循环扫描：total > capacity 的数据流水般穿过缓存，LRU 一格都留不住。开局的差异只决定谁先出事，决定不了结局。

比前篇更险恶的一点：雪崩发生时 GPU 利用率是 100%——它在满负荷地重算历史。OS 抖动的体征是 CPU 空转，一眼可诊断；KV 雪崩在利用率面板上一片健康，必须盯 goodput、命中率、TTFT 才看得见。

而且雪崩会自我放大，不需要任何人犯错：每轮变慢 → 在途请求堆积 → 池子更挤；TTFT 超时 → 客户端重试 → 同一个长 prefill 提交两遍。1968 年那个火上浇油的调度器，今天叫超时重试。

药方与要点

6. 药方还是 1968 年那一个

① Σcontext ≤ M 是生死线

一个 session 的需求 = 全部前缀，只增不减。session 数不变，需求也会自己爬过线——盯住总和，别盯个数。

② R ≈ 100 让 99% 命中率只值 50% 效率

miss 的代价是重算整个历史。miss 率必须远小于 1/R——不是洁癖，是效率公式的暴政。

③ 雪崩是隐形的

崩溃时 GPU 利用率是 100%——它在满负荷做无用功。报警要盯 goodput、命中率、TTFT，别信利用率。

④ 解法 = 准入控制

超线时让新轮次在门口排队，别放进池子互踩；逐出从 session 尾部开始；把 session 路由回缓存所在的节点。
