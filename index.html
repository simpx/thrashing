工作集与抖动
看懂 Peter Denning 的 1968

操作系统为什么会突然“卡死”？给每个程序多少内存才算够？1968 年 Denning 用两篇论文回答了这两个问题，奠定了此后几十年虚拟内存管理的理论基础。这个页面用交互式模拟带你走进这两个概念。

The Working Set Model for Program BehaviorPeter J. Denning · Communications of the ACM, 11(5), 1968 年 5 月
Thrashing: Its Causes and PreventionPeter J. Denning · AFIPS Fall Joint Computer Conference, 1968
背景

1. 分页、缺页与一个尴尬的问题

1960 年代的分时系统引入了两项关键技术：多道程序（内存里同时驻留多个进程，一个等 I/O 时另一个用 CPU）和虚拟内存分页（程序的地址空间切成固定大小的“页”，只有被用到的页才装入物理内存）。进程访问一个不在内存中的页时会触发缺页中断（page fault），必须等磁盘把页读进来才能继续——这个等待比执行一条指令慢上万倍。

于是出现了一个当时没有理论能回答的问题：操作系统应该给每个进程保留多少页的内存？给多了浪费、能同时运行的进程就少；给少了缺页频繁、进程几乎跑不动。Denning 的答案是：别拍脑袋定一个数，去测量程序自己表现出来的需求——这就是工作集。

概念一

2. 工作集：程序“正在使用”的那些页

定义　进程在时刻 t、窗口 τ 下的工作集 W(t, τ)，是它在最近 τ 次访存（时间区间 (t−τ, t] ）中引用过的不同页面的集合；|W(t, τ)| 称为工作集大小。

直觉：以“最近用没用过”作为“接下来会不会用”的估计——这背后的依据是程序的局部性（locality）：程序在一段时间内倾向于集中访问一小撮页面，然后才转移到下一撮。

可以观察到 Denning 论文中的两个核心现象：

① 稳定的平台期。在一个局部性阶段内部，工作集大小几乎恒定（约等于该阶段活跃页面数）——程序的真实内存需求是可测量、且相当稳定的。

② 阶段切换时的峰。局部性发生转移时，窗口里新旧页面并存，工作集短暂膨胀，随后旧页面滑出窗口、自然被“遗忘”。这正是工作集置换的思想：不在工作集里的页可以安全换出，无需任何额外启发式。

参数选择

3. 窗口 τ 怎么选？

τ 是工作集模型唯一的参数。τ 太小，窗口装不下一个局部性集合，正在使用的页会被误判为“不用了”，缺页率飙升；τ 太大，早已废弃的页迟迟不被释放，白占内存。Denning 证明了平均工作集大小 s(τ) 是 τ 的递增凹函数——增长先快后慢，存在一个明显的“拐点”：

定量基础

4. 缺页有多贵：效率与生存期曲线

在进入抖动之前，需要两条曲线打底——第 5 节的模拟正是这两条曲线的组合。它们不是本页的发明：《Thrashing: Its Causes and Prevention》的定量论证正是围绕它们展开的。

4.1 一次缺页 = 几千次访存的代价

先定义两个量。缺页间隔 L：进程平均连续执行多少次访存才遇到一次缺页（论文中称 lifetime，即两次缺页之间的“生存期”，按进程自己的虚拟时间计）。缺页服务时间 T：一次缺页从中断到页面调入内存所需的时间，折算成访存次数。1968 年一次内存引用约 1 µs，而磁鼓/磁盘调页约 10 ms——相差 4 个数量级。本页模拟取 T = 2000（已经是保守值）。

一个进程的 CPU 效率（Denning 称 duty factor）就是“干活时间 ÷ 总时间”：每执行 L 次访存要停下来等 T，所以

e = L / (L + T)

这条曲线解释了抖动为何是断崖而非缓坡：因为 T 大得离谱，L 稍一缩短（缺页率稍一升高），效率就贴地。要效率达到 95%，缺页间隔必须达到 19 倍 T——将近四万次访存才允许错一次。

4.2 生存期曲线：内存怎样换来缺页间隔

那么 L 由什么决定？由进程分到的内存帧数 f。L 随 f 变化的关系就是生存期曲线 L(f)，它的形状有普遍规律：在工作集大小以下近似幂律增长——多给一帧，缺页间隔成倍改善；一旦 f 覆盖了工作集则饱和——继续加内存几乎不再减少缺页。拐点（膝点）就落在工作集大小附近，这也是 Denning 后来总结的“膝点判据”。本页模拟采用：

L(f) = 20000 × (f / 15)6（当 f < 15）；　L(f) = 20000（当 f ≥ 15）

其中 15 是进程的工作集大小，20000 是饱和值。幂指数 6 是教学选择——真实程序测得的指数因程序而异（文献中常见 1.5～3），指数越大悬崖越陡，但“膝点以下幂律、以上饱和”的形状是普遍的。

概念二

5. 抖动：多塞一个进程，系统雪崩

有了工作集这把“尺子”，就能解释 1960 年代运维人员观察到的一种诡异故障：系统吞吐量随负载上升而上升，但多加一两个进程后，CPU 利用率不是缓慢下降，而是断崖式崩塌——所有进程都在等磁盘调页，CPU 几乎完全空转。Denning 把它命名为 thrashing（抖动/颠簸）。

成因　物理内存被过度分配（overcommit）：当所有进程工作集之和超过物理内存 Σ|Wi| > M，每个进程都拿不到自己工作集所需的页，于是每个进程都频繁缺页。更糟的是正反馈：CPU 空闲 → 调度器以为负载不足 → 再放入新进程 → 内存更紧张 → 缺页更多 → CPU 更闲。

每个进程分到 f = 60/N 帧 → 由生存期曲线得到缺页间隔 L(f)（第 4.2 节）→ 该进程处于“等调页”状态的时间占比 b = T / (T + L)（第 4.1 节效率的补集）。假设各进程的等待相互独立，则 CPU 空转当且仅当 N 个进程同时在等调页，概率为 bN，于是：

U(N) = 1 − bN，其中 b = 2000 / (2000 + L(60/N))

代入验证：N = 4 → f = 15，L = 20000，b ≈ 0.091，U = 1 − 0.091⁴ ≈ 99.99%；N = 10 → f = 6，L = 20000×(6/15)⁶ ≈ 82，b = 2000/2082 ≈ 0.961，U = 1 − 0.961¹⁰ ≈ 33%。

简化说明：独立性假设忽略了磁盘排队——现实中缺页请求在同一台调页设备上串行，抖动只会比模型更严重。

N ≤ 4 时（4 × 15 = 60 帧恰好用满），每个进程的工作集都装得下，加进程能填满 CPU 的 I/O 空隙，利用率接近 100%。从第 5 个进程开始，每个进程分到的帧数跌破工作集需求，缺页间隔沿第 4 节的生存期曲线崩塌，利用率从 ~99% 一路崩到接近 0——这条陡峭的悬崖就是抖动。曲线为什么先升后崩，公式说得很直白：N 小时加进程让 bN 指数级变小（用别人的计算填补 I/O 空隙）；一旦 f 跌破工作集，L 的幂律崩塌把 b 推向 1，指数再也救不回来。

解法

6. 药方：工作集原理

工作集原理（Working Set Principle）　仅当一个进程的整个工作集都在内存中时，才允许它运行；操作系统必须保证任何时刻 Σ|Wi| ≤ M。内存装不下新进程的工作集时，宁可把某个进程整体挂起（换出），也不挤压其他进程的工作集。

在上面的模拟中勾选“工作集准入控制”即可看到效果：绿色曲线在 N = 4 处封顶——多余的进程排队等待而不是挤进内存，利用率稳定在峰值平台上，悬崖消失了。用一点排队延迟，换来整个系统不崩溃。

这一原理的后代遍布现代系统：Linux 的页面回收与 refault 检测、Windows 的进程工作集管理器、数据库缓冲池的驻留集控制，乃至 CPU 缓存分析中的“工作集大小”概念，都源自这篇 1968 年的论文。

续篇　这套理论并没有留在 1968 年：《KV Cache 的工作集与雪崩》把工作集与抖动泛化到 LLM 推理的 KV cache 管理——同样的骨架，六个本质差异，以及一条更锋利的雪崩线。

带走的四个要点

工作集 = 最近 τ 次访问碰过的页

用过去预测未来，依据是程序的局部性。它是对进程真实内存需求的动态测量，而非静态猜测。

τ 选在 s(τ) 曲线拐点

覆盖一个局部性集合即可。s(τ) 是递增凹函数，拐点之后加大 τ 只是囤积陈旧页面。

抖动 = 内存过度分配的正反馈崩溃

Σ|Wᵢ| > M 时人人缺页、CPU 空转，调度器误判后火上浇油。崩溃是断崖式的，不是渐进的。

准入控制优于精细置换

保证运行中的进程工作集完整，装不下就整体挂起。控制多道程序度是防止抖动的根本手段。
