仿真设定

四种策略，同一套需求

记号：q(m) = 机器 m 当前预计排队秒数（backlog/算力）；δ = 本轮新增 token；C = 这次要复用的 context 大小；rate = 单机 token/s 处理能力；home = 索引记录的归属机器。

纯亲和：route = home（索引有效就去索引说的那台，从不因负载改路由——理论上限，也是唯一"不可能因策略而漂移"的对照组）。

朴素阈值：if q(home) > θ: route = argmin_alt q(alt) else: route = home——只看 q(home) 一个数字过没过线，θ 是滑杆上那个"敏感度"；换机器的判断里完全没出现 C，不比较代价，就是"阈值式"病灶。

TTFT 贪心：TTFT(home) = q(home) + δ/rate，TTFT(alt) = q(alt) + (δ+C)/rate；route = (TTFT(alt) < TTFT(home)) ? alt : home。把两边化简，重路由的触发条件其实是：

q(home) − q(alt) > C / rate

右边正比于 C——这次要复用的前缀越大，需要的排队落差就要越大，重路由才划算。这就是它不靠隔离也能护住别人的原因：门槛随体量自动收紧，天然只挪便宜的小请求，贵的大会话门槛高到几乎碰不到，挪的时候写进 alt 的量也小，连带驱逐自然就少（tiles 里的逐出数能直接看到这个效果）。

TTFT + 隔离：判断式一样，只是 alt 换成专门的溢出池机器——即使触发条件判断出错（比如低估了 C 的代价），溢出池也从不装任何人的常驻数据，写进去不会挤走谁。架构上把外部性摁死，不依赖"C 越大越少挪"这个自我选择去兜底。

调好参数后点击运行；命中率越低越触发得多，朴素阈值在某个临界点附近会从"缓慢下滑"突变成"瞬间崩溃"，可以自己找那个点。
