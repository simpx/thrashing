命中率会收敛吗
400 台 prefill 机器，四种调度策略，跑 90 分钟

同时用"命中率猜测 + 机器负载"做路由，在外部需求完全不变的情况下，命中率会不会自己往下漂移——这里不再是推演，是跑出来看。四种策略共享同一套需求（会话到达、节奏、寿命完全相同），只有路由决策不同。

仿真设定

四种策略，同一套需求

node = 一个简单的 block pool：只有两个接口——insert（写入，走 LRU 逐出，会挤走别人）、query（只读，告诉调用方这个会话眼下真实还留着多少前缀，不刷新 LRU、没有副作用）。调度器和 node 是分开的两个角色：node 对自己的池子是自洽的、诚实的；调度器面对的是预估的 TTFT，信息来源可以是"只看队列的盲猜"，也可以是"真调用一次 query"——这是本页唯一一个可以切换对比的开关（控制条里的"TTFT 对 home 的信念"）。

记号：q(m) = 只读接口 2 查到的机器 m 队列延迟——真实队列里，已经在处理的项用真实剩余量，还没轮到的项一律按 worst-case（全量 miss）估计，因为它们的真实代价要等真正处理到那一刻才知道；δ = 本轮新增 token；C = 这个会话到目前为止积累的全部前缀长度；rate = 单机 token/s 处理能力；home = 索引记录的归属机器。走 alt 因为索引压根没提过这台机器，缺省当没有，要算 δ+C（全量冷 miss，不建模"alt 恰好也有残留"的中间态，实测这类幽灵命中占比只有 0.1-0.2%，可以忽略）。队列的 worst-case 估计天然偏悲观——这也是这版比上一版更容易触发重路由的原因之一。

纯亲和：route = home（索引有效就去索引说的那台，从不因负载改路由——理论上限，也是唯一"不可能因策略而漂移"的对照组）。

朴素阈值：if q(home) > θ: route = argmin_alt q(alt) else: route = home——只看 q(home) 一个数字过没过线，θ 是滑杆上那个"敏感度"；换机器的判断里完全没出现 C，不比较代价，也从不 query，就是"阈值式"病灶。

TTFT 贪心：TTFT(home) = q(home) + (δ+(1−hitFrac)·C)/rate——策略1·盲猜：不 query，索引说在就假设 hitFrac=1（一定命中）；策略2·读查询：真调 node.query(id) 拿真实驻留量算 hitFrac，home 也可能已经被别的会话的写入悄悄挤掉一截，query 就能照实反映。TTFT(alt) = q(alt) + (δ+C)/rate；route = (TTFT(alt) < TTFT(home)) ? alt : home。hitFrac=1 时化简后触发条件是：

q(home) − q(alt) > C / rate

右边正比于 C，反过来读更准确：C 越小，门槛越低——小请求几乎没有免疫门槛。而 q(home) 是同一台机器上所有排队请求共享的一个数：一个贵会话的重算堆在队列里，同一时间窗口内排在它后面一串本来毫不相干的小请求都会被连坐、集体判定"该走"。两条叠加，重路由率反而高于朴素阈值（tiles 里能看到），两种策略下逐出数都远超纯亲和：算得准不等于挪得少，两种信念来源的结论一致，隔离不是可有可无的加分项。

TTFT + 隔离：判断式一样（同样可切换两种信念），只是 alt 换成专门的溢出池机器——不管上面两条效应多严重、信念来源准不准，溢出池从不装任何人的常驻数据，写进去不会挤走谁。架构上把外部性摁死，不依赖"猜得准"去兜底，这也是四条曲线里唯一稳定跟在纯亲和后面、没有掉队的一条。

调好参数后点击运行；命中率越低越触发得多，朴素阈值在某个临界点附近会从"缓慢下滑"突变成"瞬间崩溃"，可以自己找那个点。

架构进阶

如果 cache 不再跟着路由走：中心化缓存

假设变了：400 台机器不再各自维护一份缓存，而是把容量整体贡献出来，组成一个共享 LRU 池。谁在池子里留没留、留多少，只取决于全局 demand 有没有超过全局容量——跟这次请求被路由到了哪台机器完全无关。命中率不再是一份需要猜的"信息"，是物理规律的自然结果：调度器不用再猜命中率，索引、TTL 这些概念也一起不需要了。机器退化成纯粹的计算资源：只剩队列和吞吐。调度问题因此从"缓存亲和 vs 负载均衡"的取舍，缩小成一个纯负载均衡问题。

三种路由：亲和（还是那个老习惯——认准一台机器就不换，不看队列，也不看命中率反正也没用了）、随机（每次都摇一个骰子，完全不看任何信息）、最短队列（每次采样几台，挑排队最短的那台——纯负载均衡）。

一个容易漏掉、但决定了下面所有结论的建模细节：一个会话的下一轮请求，必须等这一轮真正处理完、真的收到响应之后才会发生——不是"决定路由的那一刻"就默认可以发下一句。如果不加这条约束，亲和路由下被同一台拥堵机器攒住的会话，它后续几轮会全部堆进同一条队列，排空时被背靠背连续处理，会"意外"刷出偏高的命中率——那不是真实世界的样子，只是记账方式留的漏洞。加上这条真实的反压约束之后，"总吞吐量早晚都会追平、看不出损失"这个假象也随之消失：有些机器排长队、有些机器在空转，损失是真实存在的，只是不会出现在命中率上，会出现在吞吐上。

默认参数下三条路由共享同一套需求；"单机处理速率"决定拥堵程度——调低会看到随机路由排队失稳（积压时间可能远超 90 分钟窗口，这本身就是个真实的教训：不看负载的路由，方差会失控）。

总 TPM（新 token + 重算 token，千 token/分钟）

目的：说明即使命中率已经跟路由脱钩，纯亲和这种"不看负载"的路由方式，仍然会因为排队不均而白白损失吞吐——损失不是从命中率里来的，是从"有些机器在排长队、有些机器在空转"里来的。

有效 TPM（只算新 token，千 token/分钟）

这才是"真正干了多少有用的活"：miss 掉的重算不算数，因为反压是真实的——一轮没真正处理完，下一轮就不会发生，排队不均造成的损失会直接体现在这里，不再被"总量早晚都能补上"这种假象掩盖。

最长排队积压（换算成"排到它要多久才能排空"，分钟）

全局所有机器里排队最久的那一台，如果现在不再接新请求、只吃现有积压，要多久才能排空——这是吞吐损失背后真正在发生的事：负载分布得越不均，就有机器越接近"永远排不完"。

页面中的会话到达、节奏、寿命由固定随机种子生成，四种策略共享同一份"世界演化"随机流，路由决策的随机采样单独一条流，保证四组仿真面对的是同一套需求。仅供内部推演使用，参数与真实生产环境不完全对应。
